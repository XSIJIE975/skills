# Task Decomposer

将 OpenSpec 任务清单细化为可执行的详细任务计划，分析依赖关系，分配优先级。

## 输入

- OpenSpec 变更目录路径
- `tasks.md`（任务清单）
- `specs/requirements.md`（需求规格）
- `design.md`（技术设计）

## 输出

- 细化后的任务计划（每个任务包含详细步骤、验收标准、依赖、预估时间）
- 依赖关系图（识别可并行的任务组）
- 更新后的 `tasks.md`

## 流程

### 1. 读取和分析任务

- 读取 `tasks.md`，提取所有任务条目
- 读取 `specs/requirements.md` 和 `design.md` 作为上下文
- 识别不明确的任务：描述含 "TBD"、"待定"、"TODO"；缺少验收标准；范围过宽或过窄

### 2. 澄清模糊需求

对不明确的任务，调用 brainstorming skill 进行交互式澄清：
- 提出具体澄清问题
- 根据用户回答完善任务描述
- 补充验收标准和技术约束

所有任务清晰后进入下一步。

### 3. 任务细化

为每个任务生成：

```
## 任务：<标题>

### 验收标准
- [ ] 标准 1
- [ ] 标准 2

### 实现步骤
1. 具体步骤...
2. 具体步骤...

### 依赖
- 前置任务：<任务编号或 "无">

### 预估时间
- <X> 小时

### 测试要求
- 单元测试：<描述>
- 集成测试：<描述（如适用）>
```

### 4. 依赖分析和并行识别

- 构建任务依赖 DAG（有向无环图）
- 识别无依赖的任务组 → 可并行执行
- 标记关键路径（最长依赖链）
- 输出执行建议：先执行哪些并行任务，再串行执行依赖任务

### 5. 写回任务清单

将细化后的任务更新到 `tasks.md`，保留原始任务结构，补充详细信息。

## 约束

- 每个任务的预估时间不超过 4 小时（超过则拆分）
- 测试要求必须与验收标准对应
- 依赖关系不能有环
- 优先级由依赖深度决定（无依赖 = 高优先级）
