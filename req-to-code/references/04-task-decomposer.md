# Task Decomposer

将 OpenSpec 已确认的任务清单细化为可执行的详细任务计划，分析依赖关系，分配优先级。

对于细化过程中发现的模糊需求，提供两级澄清机制：
1. **简单澄清**：A/B/C 选择题，适用于只需补充少量信息即可明确的场景
2. **深入探讨对话**：一问一答、提出 2-3 种方案供用户选择，适用于真正模糊/需探讨多种方案的场景

## 输入

- OpenSpec 变更目录路径
- `tasks.md`（已通过阶段 3 门禁确认的任务清单）
- `specs/requirements.md`（已确认的需求规格）
- `design.md`（已确认的技术设计）

## 输出

- 细化后的 `tasks.md`（每个任务包含详细步骤、验收标准、依赖、复杂度、测试要求）
- 依赖分析（内联在 tasks.md 汇总部分：依赖关系、关键路径、可并行的任务组）

## 流程

### 1. 读取和分析任务

- 读取 `tasks.md`，提取所有任务条目
- 读取 `specs/requirements.md` 和 `design.md` 作为上下文
- 识别不明确的任务：
  - 描述含 "TBD"、"待定"、"TODO"
  - 缺少验收标准
  - 范围过宽或过窄
  - 与需求规格/技术设计存在歧义

### 2. 模糊任务处理决策树

对每个模糊任务按以下决策树处理：

```
任务是否模糊？
  │
  ├─ 只是缺细节（验收标准不完整、描述太简单）
  │   └─→ 走"简单澄清"（A/B/C 选择题）
  │
  ├─ 存在多种方案不确定选哪个
  │   └─→ 走"深入探讨对话"
  │
  ├─ 与已确认的 design.md 有冲突/需补充设计决策
  │   └─→ 走"深入探讨对话"
  │
  └─ 用户之前已确认过（阶段 3 门禁中确认了类似内容）
      └─→ 不提问，按已确认的内容直接细化
```

### 3. 简单澄清（A/B/C 选择题）

适用于只需补充少量信息的任务：

```
任务 "<任务名>" 的验收标准不够明确，请确认：
A. <方案 A>（推荐，自动生成的建议）
B. <方案 B>
C. 自定义（请说明）
```

**规则**：
- 每次只问 1 个问题
- 问题必须是选择题（A/B/C）
- 用户选择后，自动更新任务描述
- 如果用户的选择需要级联更新其他任务，自动处理，不额外提问
- 如果用户对 A/B/C 都不满意选择了自定义，但描述仍然模糊 → 升级为深入探讨对话

### 4. 深入探讨对话

适用于真正模糊、需要探讨多种方案的场景。

调用 superpowers 的 **brainstorming** 技能进行讨论，但用 prompt 限定其 scope——只要对话能力，不要文件产出。

加载时限定以下约束（Agent 根据约束自行构造 prompt）：
- 只讨论当前这 1 个模糊点，不探索项目整体上下文
- 用一问一答模式提出 2-3 种方案供用户选择
- **不要**将讨论结果写入 docs/superpowers/ 目录
- **不要**执行 spec self-review 或调用 writing-plans
- 讨论结束后，将结论返回给调用者，由调用者更新到 OpenSpec 文档中

| 做 | 不做 |
|---|------|
| ✅ 一问一答，每次只问 1 个问题 | ❌ 不写设计文档到 `docs/superpowers/specs/` |
| ✅ 提出 2-3 种方案 + 推荐建议 | ❌ 不走 spec self-review 或 writing-plans |
| ✅ 聚焦**当前这一个**模糊点 | ❌ 不探索项目上下文（阶段 3 已完成） |
| ✅ 讨论完提取结果更新 OpenSpec 文档 | ❌ 不创建独立 plan 文件 |
| ✅ 讨论完毕即结束，回到本阶段正常流程 | |

用户选择后，追问不超过 2 轮以收敛细节。然后：

**结果提取和回填**：
- 将对话结论提取为结构化信息
- 更新 `tasks.md` 中对应任务的描述、验收标准
- 如果影响到了 `specs/requirements.md` 或 `design.md`，同步更新（版本号 +1，标注为 "深入探讨补充"）
- 在状态文件的 `user_feedback` 中记录对话摘要

**⚠️ 门禁保护规则**：
如果深入探讨的结果**修改了 requirements.md 或 design.md 的核心内容**（目标、验收标准、架构选型、技术栈），则：
1. 更新对应文档
2. **标记阶段 3 中相应的门禁为 `needs_revalidation`**
3. 回到阶段 3 的门禁重新确认（只展示 diff）
4. 重新确认通过后，再继续本阶段

这防止本阶段悄悄绕过阶段 3 已确认的内容。

**边界保护**：
- 每次对话只讨论 **1 个**模糊点
- 最多对话 **3 轮**（1 轮提方案 + 最多 2 轮追问），超过则转为"记录为待定，升级给用户决策"
- 讨论结束后**必须显式结束对话**，回到任务细化流程
- 不创建任何 `docs/superpowers/` 目录下的文件

### 5. 任务细化

所有模糊任务澄清完毕后，为每个任务生成细化内容，写入 `tasks.md`：

```markdown
## 任务：<标题>

### 验收标准
- [ ] 标准 1
- [ ] 标准 2

### 实现步骤
1. 具体步骤...
2. 具体步骤...

### 依赖
- 前置任务：<任务编号或 "无">

### 复杂度
- 低 / 中 / 高
- 判断依据：涉及文件数、逻辑复杂度、外部依赖数

### 测试要求
- 单元测试：<描述>
- 集成测试：<描述（如适用）>
```

### 6. 依赖分析和并行识别

- 构建任务依赖 DAG（有向无环图）
- 识别无依赖的任务组 → 可并行执行
- 标记关键路径（最长依赖链）
- 输出执行建议：先执行哪些并行任务，再串行执行依赖任务

### 7. 汇总确认（1 次，结构化）

细化完成后，向用户做**一次性汇总**展示：

```
任务细化完成：
- 共 <N> 个任务，<M> 个可并行
- 关键路径：<依赖链>
- 共通过 <简单澄清次数> 次简单确认 + <深入探讨次数> 次对话讨论了模糊点

请确认：
1. 任务分解和依赖关系是否正确？
2. 任务复杂度评估是否合理？
3. 是否可以开始执行？
```

用户确认后，最终的 `tasks.md` 是阶段 5（Intelligent Orchestrator）的执行依据。

### 8. 输出执行计划

细化后的 `tasks.md` 是阶段 5（Intelligent Orchestrator）的唯一执行依据。
不创建独立的 plan 文件到 `docs/superpowers/plans/` 或类似目录。
所有执行所需信息（实现步骤、测试代码、依赖关系）全部内联在 tasks.md 中。

## 约束

- 每个任务的复杂度不超过"高"（达到"高"则需拆分为多个子任务）
- 测试要求必须与验收标准对应
- 依赖关系不能有环
- 优先级由依赖深度决定（无依赖 = 高优先级）
- 深入探讨对话每次只讨论 **1 个**模糊点，不超过 3 轮
- 深入探讨后**必须**将结果更新到 OpenSpec 文档中，不允许结果只停留在对话历史
- 汇总确认只做 1 次，不重复确认
