# Archive Manager

归档完成的变更，生成交付报告，清理资源。

## 输入

- OpenSpec 变更目录路径
- 审查报告（来自 deep-review-loop）
- 执行报告（来自 intelligent-orchestrator）
- 代码变更统计

## 输出

- 交付报告（`archive/<change-name>/delivery-report.md`）
- 归档的变更目录
- 清理后的项目状态

## 流程

### 1. 归档前验证

必须全部通过才允许归档：

| 检查项 | 验证方式 | 不通过处理 |
|--------|---------|-----------|
| 所有任务完成 | 检查 `tasks.md` 中无 `- [ ]` 项 | 阻塞归档，列出未完成任务 |
| 审查通过 | 检查 `review-report.md` 无 "未通过" | 阻塞归档，提示重新审查 |
| 测试通过 | 运行项目测试命令，检查退出码 | 阻塞归档，提示修复测试 |

### 2. 生成交付报告

创建 `delivery-report.md`，包含：

```markdown
# 交付报告

## 变更摘要
- 变更名称：<name>
- 变更类型：<feature/bugfix/refactor>
- 完成时间：<ISO-8601>

## 需求覆盖
- 需求规格：<path>
- 覆盖率：100%（每个需求对应一个或多个任务）

## 交付物
- 代码变更：<N> 个文件，+<X> / -<Y> 行
- 测试：通过 <M> 个测试用例
- 覆盖率：<Z>%（如可用）

## 质量审查
- 审查轮数：<N>
- 严重问题：<M> 个（已全部修复）
- 重要问题：<K> 个（已修复 / 记录技术债）

## 后续建议
- <建议 1>
- <建议 2>
```

### 3. 执行归档

- 将变更目录移至 `openspec/changes/archive/<change-name>/`
- 清理 `/tmp/req-to-code/` 临时文件
- 删除 `.req-to-code-state.json`
- 更新 `openspec/openspec.json` 中的归档记录

### 4. 完成通知

向用户报告：
- 归档完成
- 交付报告路径
- 变更摘要（文件数、行数、测试数）

## 约束

- 归档前验证不可跳过
- 交付报告必须包含可量化的数据
- 临时文件必须清理
- 状态文件在归档成功后删除
